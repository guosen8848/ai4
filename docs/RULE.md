## 2. 前后端结构约定


### 2.1 后端结构约定

后端原则上采用 `api / service / crud` 三层结构。

* `api`：负责接口定义、请求参数接收、响应返回
* `service`：负责业务逻辑、流程编排、权限判断、参数校验
* `crud`：负责数据库操作，不编写业务逻辑

可选目录：仅在实际需要或用户确认后创建

* `schemas`：负责请求和响应数据结构定义
* `models`：负责数据库模型定义
* `core`：负责核心配置、安全认证、异常处理等
* `utils`：负责通用工具函数

### 2.2 前端结构约定

前端原则上采用以下结构：

* `api`：封装后端请求
* `router`：管理路由
* `layouts`：页面整体布局
* `views`：页面级组件
* `components`：通用 UI 组件
* `composables`：通用组合式逻辑
* `utils`：通用工具函数
* `stores`：状态管理
* `constants`：常量定义

## 3. 接口约定

### 3.1 基础规则

* 后端接口统一以 `/api/v1` 开头
* 接口路径使用小写字母和中划线
* 请求和响应统一使用 JSON
* 时间字段统一使用 ISO 8601 格式
* 列表接口统一支持分页
* 接口返回结构保持一致
* 新增接口必须补充到接口调用链路中

### 3.2 统一响应格式

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

### 3.3 分页响应格式

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 10
  }
}
```

### 3.4 错误响应格式

```json
{
  "code": 400,
  "msg": "error message",
  "data": null
}
```

## 4 字段命名约定

* 数据库字段使用小写字母和下划线
* Python 变量使用小写字母和下划线
* 前端变量使用 camelCase
* 接口 JSON 字段原则上与后端 Schema 保持一致
* 时间字段统一使用 `created_at`、`updated_at`
* 主键字段统一使用 `id`

## 5. AI 编码注意事项

AI 在编码时必须遵守：

* 新增功能前先确认功能属于哪个模块
* 新增接口后必须更新接口清单和调用链路
* 新增页面后必须更新页面清单
* 新增数据表后必须更新数据模型约定
* 不允许绕过 `api / service / crud` 分层
* 不允许在 `api` 层直接写数据库操作
* 不允许在 `crud` 层写业务逻辑
* 不允许无理由大规模重构目录结构
* 不允许为了“更完整”主动扩展当前需求范围
