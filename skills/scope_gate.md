
# Skill: 最小功能实现检查

## 最小功能判断标准

一个需求只有同时满足以下条件，才可以认为是最小功能实现：

- 只解决一个明确问题
- 只有一个主要业务闭环
- 可以独立验收
- 不依赖大量前置功能
- 不包含多个角色的复杂协作
- 不包含多个页面或多个子系统
- 不包含明显可以后置的高级功能
- 当前实现失败后，不会影响大范围已有功能

## 非最小功能特征

如果需求出现以下情况，通常判断为不是最小功能：

- 一个需求里包含多个功能点
- 同时涉及新增、修改、删除、查询、统计、导入、导出
- 同时涉及前端、后端、数据库、权限、部署、监控
- 同时包含多个角色，例如管理员、老师、学生、审核员
- 需要设计完整系统
- 需要一次性实现完整业务流程
- 包含“顺便支持”“同时实现”“完整支持”“通用化”“可配置化”
- 第一版就包含高级功能

## 目标

当用户提出功能需求、开发计划、更新计划或系统设计时，先判断该需求是否属于“最小功能实现”。

如果判断结果不是最小功能实现，必须暂停继续执行，不继续写代码、不继续设计详细方案、不继续拆完整架构，而是指出问题并提出缩小范围的建议。

## 适用场景

当用户提出以下类型需求时使用：

- 新增一个功能
- 开发一个系统
- 修改一个模块
- 编写执行计划
- 让 AI 根据需求改代码
- 让 AI 设计接口、数据库、页面或业务流程

## 核心规则

1. 在执行任何详细设计、写代码、改文件之前，先做最小功能判断。
2. 如果需求已经是最小功能，可以继续执行。
3. 如果需求不是最小功能，必须暂停执行。
4. 暂停后，只允许输出判断结果、问题说明和缩小建议。
5. 不允许在需求过大的情况下继续补全完整方案。
6. 不允许主动扩展权限、多租户、统计、审批、消息通知、复杂配置、监控、大屏、导入导出等非必要能力。
7. 每次只推进一个最小闭环。


## 执行流程

### 第一步：复述当前需求

用一句话说明用户当前想做什么。

### 第二步：判断是否为最小功能

输出：

- 判断结果：是 / 否
- 原因：简要说明

### 第三步：如果是最小功能

继续执行用户原本要求。

### 第四步：如果不是最小功能

必须暂停执行。

只输出以下内容：

1. 为什么它不是最小功能
2. 哪些内容范围过大
3. 建议缩小成哪个最小功能
4. 需要用户确认的问题

不得继续输出完整设计方案。

## 输出格式


```text
## 最小功能判断

判断结果：是

原因：
xxx

```
